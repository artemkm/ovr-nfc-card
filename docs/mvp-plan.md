# MVP NFC-карточек членов ОВР

## Цель MVP

Собрать демонстрационный MVP онлайн-визитки члена Общества врачей России.

Основной сценарий:

```text
NFC-карта -> телефон -> URL -> публичная страница участника
```

На первом этапе проект делается для красивой и быстрой демонстрации. Финальная промышленная версия может быть позже реализована заново на Битриксе, поэтому в MVP не закладываем миграцию старых технических ссылок.

## Что фиксируем для первого этапа

- Используем технический домен Yandex API Gateway.
- Основной сайт ОВР и Битрикс пока не трогаем.
- На NFC записываем только полный URL.
- Данные пользователя на NFC не записываем.
- UID NFC-метки не используем как секрет.
- Карты на MVP не блокируем в read-only.
- Для теста подходят NTAG215, для боевого процесса планируются NTAG213 NXP.

## Формат членских номеров

Канонический членский номер в коде и базе хранится полностью:

```text
OVR-0000-0000
```

Префикс `OVR` храним как часть номера, потому что это устойчивый
идентификатор члена общества, а не только оформление интерфейса.

На пластиковой NFC-карте печатается только цифровая часть:

```text
0000 0000
```

Печатный номер карты является производным от `member_number` и отдельно в базе
не хранится. Например, `OVR-0000-0001` печатается как `0000 0001`.

Пример будущего технического URL:

```text
https://<gateway-id>.apigw.yandexcloud.net/c/{token}
```

Локально этот же сценарий будет проверяться через:

```text
http://localhost:3000/c/{token}
```

## Локальная проверка

Перед Yandex Cloud сделан локальный прототип на Node.js без внешних npm-зависимостей.

Локальный сценарий:

```text
браузер -> http://localhost:3000/admin -> локальный Node.js сервер -> тестовые данные
```

Что можно проверить локально:

1. Открыть админку.
2. Войти под локальным логином и паролем.
3. Увидеть список участников в порядке добавления в базу.
4. Листать участников страницами по 50 записей.
5. Найти врача по подстроке.
6. Увидеть, есть ли у него NFC.
7. Открыть карточку врача.
8. Сгенерировать NFC-ссылку.
9. Скопировать готовый URL.
10. Открыть публичную карточку по URL.
11. Проверить перегенерацию токена.

Локальные тестовые данные хранятся в `data/sample-members.json`. При первом запуске создается рабочий файл `data/local-db.json`; он не коммитится и используется только для локальных проверок.

Для записи на реальную NFC-карту понадобится уже публичная HTTPS-ссылка из Yandex Cloud, потому что телефон не сможет открыть `localhost` на компьютере.

## Таблицы

### member_profiles

```text
member_number          членский номер в формате OVR-0000-0000, primary key
full_name              ФИО
specialty              специальность
scientific_title       научное звание, если указано
academic_degree        научная степень, если указана
position               должность
city                   город
public_phone           публичный телефон, если разрешено
public_email           публичный email, если разрешено
is_active_member       активен ли член общества
created_at             дата создания
updated_at             дата обновления
```

Поля `region` и `avatar_url` на первом этапе не используем.

### member_cards

```text
member_number          связь с профилем
token                  случайный публичный токен
public_url             полный готовый URL для записи на NFC
qr_svg                 SVG QR-код, ведущий на public_url
status                 active, blocked, reissued
nfc_uid                опционально, если захотим учитывать UID метки
created_at             дата выпуска
updated_at             дата обновления
last_seen_at           последнее сканирование
scan_count             счетчик открытий
```

Для MVP у одного пользователя одна актуальная NFC-карта. При перегенерации старый токен заменяется новым, старая ссылка перестает работать.

## Публичная страница

Endpoint:

```text
GET /c/{token}
```

Поведение:

- токен не найден: показать страницу "карта не найдена";
- карта найдена, но статус не `active`: показать "карта недействительна";
- карта активна: показать публичную визитку;
- при успешном открытии увеличить `scan_count`;
- обновить `last_seen_at`.

Публичная визитка показывает только данные, которые можно демонстрировать публично:

- ФИО;
- членский номер;
- номер на пластиковой карте;
- научное звание, если не заполнено — прочерк;
- научная степень, если не заполнена — прочерк;
- должность;
- город;
- публичный телефон, если есть;
- публичный email, если есть;
- статус действующего члена общества;
- QR-код визитки, ведущий на эту же публичную страницу.

Специальность показывается в админской карточке и списке, но не выводится в публичной визитке.

## Админка

Технический адрес в Yandex Cloud:

```text
https://<gateway-id>.apigw.yandexcloud.net/admin
```

Локальный адрес:

```text
http://localhost:3000/admin
```

На MVP админка простая, техническая, с логином и паролем.

### Сценарий интерфейса

1. Пока пользователь не выбран, показывается экран поиска и список участников.
2. Список участников идет в порядке добавления в базу.
3. На странице списка показывается до 50 участников.
4. На широком экране пагинация находится рядом с пояснением списка.
5. На узком экране сначала идет пояснение, затем номер страницы, затем кнопки "Назад" и "Вперед" по краям блока.
6. Поиск работает по подстроке.
7. Поиск динамический: данные подтягиваются при вводе.
8. Искать можно по ФИО, членскому номеру, специальности, должности и городу.
9. В списке каждый пользователь кликабельный.
10. В списке видно, есть у пользователя NFC или нет.
11. После клика открывается карточка пользователя.
12. В карточке сверху есть кнопка "Назад к поиску".
13. Если NFC уже есть, показывается готовая ссылка и кнопка "Скопировать".
14. Если NFC нет, показывается кнопка "Сгенерировать NFC".
15. Если NFC уже есть, показывается кнопка "Перегенерировать".
16. При перегенерации показывается подтверждение: "Старый номер будет стерт, вы уверены?"
17. Если выбрать "Нет", ничего не меняется.
18. Если выбрать "Да", генерируется новый токен, обновляется `public_url` и пересоздается QR-код.

Интерфейс локального прототипа оформлен под стиль ОВР: логотип в шапке, основной синий цвет `#0076bd`, основные кнопки с белым текстом на синем фоне. Красные destructive-кнопки остаются красными, метка `NFC есть` остается зеленой, `NFC нет` остается серой.

## API MVP

Публичный endpoint:

```text
GET /c/{token}
```

Админские endpoints:

```text
GET  /admin
POST /admin/login
POST /admin/logout
GET  /admin/api/members?page={page}
GET  /admin/api/search?q={query}
GET  /admin/api/members/{member_number}
POST /admin/api/members/{member_number}/card/generate
POST /admin/api/members/{member_number}/card/regenerate
```

## Генерация токена

Нельзя использовать:

- членский номер;
- короткий последовательный id;
- UID NFC-метки.

Используем криптостойкий random:

```js
crypto.randomBytes(24).toString('base64url')
```

Готовая ссылка собирается из:

```text
BASE_PUBLIC_URL + "/c/" + token
```

И сохраняется в `member_cards.public_url`.

QR-код генерируется автоматически по `public_url` при выпуске или
перегенерации NFC-ссылки и сохраняется в `member_cards.qr_svg`.

## GitHub

GitHub уже подключен:

```text
git@github.com:artemkm/ovr-nfc-card.git
```

Текущая основная ветка:

```text
main
```

Порядок был таким:

1. Создать GitHub-репозиторий `artemkm/ovr-nfc-card`.
2. Создать отдельный SSH-ключ для проекта.
3. Добавить public key в GitHub.
4. Закрепить ключ в локальном Git config через `core.sshCommand`.
5. Добавить remote `origin`.
6. Сделать первый commit.
7. Запушить проект на GitHub.
8. Использовать GitHub как основное хранилище кода.

Если SSH-ключа нет, процесс будет такой:

1. Проверить существующие SSH-ключи.
2. Создать новый ключ для GitHub.
3. Добавить public key в настройки GitHub.
4. Проверить подключение командой `ssh -T git@github.com`.
5. Подключить remote.
6. Выполнить первый push.

Коммиты и push выполняются только после отдельного подтверждения.

## Yandex Cloud

После локальной проверки:

1. Создаем YDB-таблицы.
2. Адаптируем хранилище с локальных данных на YDB.
3. Создаем Yandex Cloud Function.
4. Настраиваем API Gateway.
5. Получаем технический домен.
6. Задаем `BASE_PUBLIC_URL` равным техническому домену.
7. Проверяем публичные и админские маршруты.
8. Записываем полученный `public_url` на NFC-карту.
